# v1.0.0 — Typed decision lane, plans under roadmap, migration registry

**First explicitly versioned release.** The major bump reflects the breaking
PM-folder layout change. If you're upgrading from a pre-v1.0.0 install, see
the **Migrating from pre-v1.0.0** section below — the agent will offer to
run the migration on your next PM-skill interaction, or you can run it
manually.

## What's new

- **`decisions/` is now a typed, first-class PM lane at the project root.**
  Records decisions *made* with seven type codes: `ADR` (architecture),
  `PRD` (product), `MKT` (market/positioning), `VND` (vendor pick),
  `POL` (policy), `NEG` (explicit rejection), `EXP` (time-boxed experiment).
  ADRs are one type, not the only kind.
- **`roadmap/plans/` is the canonical home for scoped plans.** Plans are
  mirrored into `roadmap/done-pending.md` when in flight; deferred items
  have a single source of truth.
- **Declarative migration registry.** Adding a new migration is one new file
  in `scripts/migrations/` plus one line in `_index.mjs`. The runner,
  validator, and CLI stay the same.
- **First registered migration: `1.0.0-lane-restructure`.** Restructures
  `planning/` → `roadmap/plans/`, promotes `planning/decisions/` to a
  top-level `decisions/`, renames `ADR-NNN_*.md` → `D-NNN_ADR_*.md`,
  rewrites frontmatter (`pageType: adr` → `pageType: decision`), and
  rewrites wikilinks.
- **Agent-driven migration triggers.** When `check-pm.mjs` flags an
  unapplied registered migration for the project you're working on, the
  agent offers to run it once per session — names the specific migration,
  lists its effects, and asks before mutating.
- **`install.sh --channel v1`** and **`--ref v<X.Y.Z>`** for versioned
  installs. The default still resolves to `main` in this release; a
  follow-up release flips the default to the `v1` channel.

## Breaking changes

- **`planning/` (top-level) is gone.** Use `roadmap/plans/`.
- **`planning/decisions/` (top-level) is gone.** Use `decisions/` at the
  project root.
- **ADRs are no longer a separate artifact class.** They are typed
  decisions (`decision_type: ADR`). Filename `ADR-NNN_<slug>.md` becomes
  `D-NNN_ADR_<slug>.md`. Frontmatter `pageType: adr` becomes
  `pageType: decision`.
- **`update.sh` is deleted.** Use `install.sh --update` instead.
- **`templates/ADR.md` is replaced by `templates/decision.md`.**

## Migrating from pre-v1.0.0

The agent will offer to run the migration on your next PM-skill interaction
("migrate this project", "verify setup", "log this" — any intent that runs
`check-pm.mjs`). It will name the specific migration, list its effects, and
ask once per session before mutating.

To run the migration manually:

```bash
node <skill_dir>/scripts/migrate.mjs --project <Name> --yes
```

Or for a path you don't have in `projects.json`:

```bash
node <skill_dir>/scripts/migrate.mjs --pm-folder <path-to-PM-folder> --yes
```

Add `--dry-run` first to see the plan. Add `--list` to see all registered
migrations. The runner is idempotent; re-running on an already-migrated
project is a no-op.

The migration preserves `archive/` and `history/` untouched. If you have
old wikilinks pointing from those folders at moved plans, they'll surface
as unresolved links in `check-pm-consistency.mjs` — which is the correct
behavior; decide per-link whether to add a redirect note, fix the wikilink,
or leave it for historical accuracy.

After the migration runs, append the suggested history bullet (printed by
the script) to `history/<YYYY-MM>/<date>.md`.

```bash
# Pin to v1.0.0 exactly:
```
